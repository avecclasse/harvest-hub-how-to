Introduction & Concepts

Get oriented and learn how web apps work before you build yours.

Welcome

Congratulations on starting this journey. By the end of this tutorial, you will have a real, working web application on the internet—one that other people can visit and use.

You don’t need any coding experience. You’ll be directing an AI assistant (inside an editor called Cursor) to write the code for you. Your job is to follow the steps, paste the right prompts, and make decisions about your app. The AI does the heavy lifting.

That might feel surprising at first, but it’s a powerful way to learn: you see what gets built, you can change it by asking in plain English, and you build a mental model of how apps work along the way.

What We're Building

You’re going to build Harvest Hub: a customer-facing web app where people can reserve a weekly produce bundle, choose a pricing tier that supports the community (Equity, Anchor, or Steward), learn about the mission behind the food system, and track their order as it moves through a simple allocation process.

The app includes a community capacity pool dashboard (bundles funded, households supported, progress toward distribution), a “Learn” section that explains why the system exists and where produce comes from, and order tracking with clear statuses—from “Order Received” to “Ready for Pickup.”

This is a real app. Once it’s deployed, you can share the URL and others can create accounts, make reservations, and see their orders. There’s no real payment in this version—it’s reservation-only—but the experience is complete and usable.

Understanding Dynamic Web Apps

What is a Web App?

A website is usually a set of static pages: text, images, and links that don’t change based on who’s viewing or what they do. Think of a restaurant’s menu page—everyone sees the same thing.

A web app is interactive and can store data. Users can sign in, create and save things, and see content that’s specific to them. Google Docs is a web app; so is your bank’s site. Harvest Hub is a web app because users have accounts, save their reservations, and see their own order history.

The Client–Server Model

Web apps work like a restaurant:

  • The dining room is where customers sit, read the menu, and place orders. They only see and interact with what’s in front of them.
  • The kitchen is where food is prepared and stored. Customers never go into the kitchen.
  • The waiter carries orders from the dining room to the kitchen and brings the food back.

In web terms:

  • Frontend (client) = The dining room. It’s what users see in their browser—buttons, forms, text, images. It runs in the user’s device.
  • Backend (server) = The kitchen. It stores data, processes requests, and sends responses. Users don’t see it directly.
  • API = The waiter. It carries requests from the frontend to the backend and brings responses back.

Why Do We Need Both?

  • Data persistence: If data only lived in the browser, it would disappear when the tab is closed. The backend stores it so it’s there next time.
  • Multiple users: If two people use your app, they need to see consistent data. That data has to live in one central place—the backend.
  • Security: Some logic (like “only this user can see their orders”) must run where users can’t change it—on the backend.

What This Means for Your App

For Harvest Hub, the frontend is built with React (through a framework called Next.js)—a standard way to build interactive interfaces. The backend—data storage and user accounts—is handled by InstantDB, a service that gives you a database and authentication without building a server yourself. So you get the benefits of the client–server model without writing backend code from scratch.
